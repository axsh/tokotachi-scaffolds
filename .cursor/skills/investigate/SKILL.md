# 調査 (Investigation)

コードベースの調査・分析を行い、結果をレポートとしてまとめる。
**コードの修正や変更は一切行わない。**

## 目的

- バグの原因特定、設計の理解、依存関係の把握などの調査を体系的に実施する
- 調査結果を構造化されたレポートとしてまとめ、ユーザーに報告する
- コードベースに影響を与えず、読み取り専用で作業する

## 制約事項

> [!CAUTION]
> - ソースコードの変更（追加・編集・削除）は**絶対に行わない**
> - `git commit`, `git push` 等のリポジトリ変更操作は**禁止**
> - ビルドスクリプトや生成系コマンドの実行は**禁止**（調査目的の読み取り系コマンドは可）

### 許可される操作

- ファイルの閲覧・検索（grep, find, cat, head, tail 等）
- `git log`, `git blame`, `git diff`, `git show` 等の読み取り系Gitコマンド
- 既存のバイナリやコマンドの実行（現状把握のため。例: `--version`, `--help`, ステータス確認系）
- ログファイルの閲覧・分析
- `go doc`, `go vet`, `go list` 等の読み取り系Goツール
- 中間ファイルの出力（`tmp/` ディレクトリ以下のみ）

### 禁止される操作

- ソースコード（`.go`, `.ts`, `.tsx`, `.css`, `.html` 等）の作成・編集・削除
- 設定ファイル（`.json`, `.yaml`, `.toml` 等）の作成・編集・削除
- 仕様書・実装計画書の作成・編集（`prompts/` 配下）
- `go build`, `go install`, `npm run build` 等のビルドコマンド
- テストの実行（`go test`, `scripts/process/build.sh`, `scripts/process/integration_test.sh` 等）

## 1. 調査の開始

1. **調査対象の確認**:
    * ユーザーの依頼内容を正確に把握する。
    * 調査の目的（バグの原因特定、設計理解、パフォーマンス分析など）を明確にする。
    * 不明点がある場合は、調査前にユーザーに確認する。

2. **調査スコープの決定**:
    * 調査対象のファイル、モジュール、機能を特定する。
    * 必要に応じて、関連する仕様書（`prompts/phases/` 配下）や設計ドキュメントを参照する。

## 2. 調査の実施

以下の手法を適宜組み合わせて調査を進める:

1. **コード解析**:
    * ソースコードの閲覧と理解
    * 関数・構造体の依存関係のトレース
    * grep/ripgrep によるパターン検索
    * `git blame` / `git log` による変更履歴の追跡

2. **ログ・出力の分析**:
    * アプリケーションログの確認
    * エラーメッセージやスタックトレースの分析
    * 設定値の確認

3. **動的な現状把握**（必要に応じて）:
    * 既存のコマンドやバイナリの実行によるバージョン・ステータス確認
    * 環境変数や設定の確認
    * プロセスやサービスの状態確認

4. **ドキュメント・仕様の参照**:
    * 既存の仕様書（`prompts/phases/` 配下）の確認
    * コード内のコメントや GoDoc の確認
    * README や設計ドキュメントの確認

## 3. レポートの作成

調査結果をレポートにまとめる。

### レポートの構成

1. **調査概要 (Investigation Summary)**:
    * 調査の目的と背景
    * 調査対象のスコープ

2. **調査手法 (Methodology)**:
    * 実施した調査の手法を簡潔に記載
    * 使用したコマンドや検索パターンなど

3. **調査結果 (Findings)**:
    * 発見した事実を構造化して記載
    * コードスニペット、ログ抜粋、図表などを活用
    * ファイルパスへのリンクを含める

4. **分析・考察 (Analysis)**:
    * 調査結果に基づく分析と考察
    * 原因の推定や問題の構造の説明
    * 関連する既知の問題や注意点

5. **推奨事項 (Recommendations)**（該当する場合）:
    * 調査結果を踏まえた対応の提案
    * 優先度の提示
    * **注意**: 方向性や方針の提案のみ。実際のコード変更は行わない。

### レポートのフォーマット

- マークダウン形式で記述する
- コードスニペットはコードブロックで記載する
- ファイルパスへのリンクを含める（プロジェクトルートからの相対パス）
- 必要に応じて Mermaid 図やテーブルを使用する

## 4. 報告

1. **レポートの提示**:
    * 作成したレポートをユーザーに提示する。
    * 重要な発見事項がある場合は、チャットでもハイライトする。

2. **フォローアップ**:
    * ユーザーからの追加質問に対応する。
    * 必要に応じて追加調査を実施する。
    * 修正が必要な場合は、別スキル（`create-specification` → `create-implementation-plan` → `execute-implementation-plan`）の利用を提案する。
